What Georgia Kept

One afternoon with DPLA's bulk export and an AI that kept hitting CORS errors

Sara Palmer · Emory Center for Digital Scholarship (ECDS)

01Introduction

1.1 million records, one afternoon

The Digital Public Library of America aggregates metadata from hundreds of contributing institutions. In March 2026, the Digital Library of Georgia hub alone contained 1,149,945 records — photographs, maps, documents, sound recordings, and moving images contributed by 113 different institutions across the South and beyond.

What does a collection of that size actually look like? Not what it contains — what it is: its shape over time, its distribution across format types, its gaps and concentrations. These are questions that metadata can answer, but only if you look at it in aggregate rather than record by record.

This report uses the DPLA bulk export to explore three questions: How has the format composition of this collection shifted over time? How long did items wait between their creation and their digitization — and does that lag vary by institution type? And what do a handful of individual items look like when pulled back out of the aggregate?

All data comes from the DPLA March 2026 bulk export for the Georgia hub, processed locally with Python and visualized with D3.js. See the methodology note at the bottom for details on tools and process, including where an AI assistant helped and where it ran into walls.

02The shape of a collection

What format is history?

Of the 157,402 dated items in the collection, the vast majority are images — photographs, maps, illustrations. But the mix shifts dramatically across time, and the shifts carry meaning.

The 1864 spike in text records reflects the Civil War: newspapers, orders, correspondence, and official documents, concentrated in a single year. The dominance of images from 1930 through 1960 corresponds to the FSA photography programs and the era of institutional photography. Sound and moving image arrive late — essentially from scratch in the late 1950s — tracing the moment when the Peabody Awards collection enters the archive.

The gold overlay tracks items tagged with civil rights or African American subject headings. Its arc — building slowly through Reconstruction, peaking sharply in 1965 and again around 1970–72 — reflects both history and the retrospective work of archivists deciding what to describe and how.

03The waiting room

How long did things wait?

Every item in DPLA has two dates embedded in its metadata: when it was created, and when it was ingested into the digital library. The gap between those dates — the digitization lag — is a rough proxy for institutional priority.

The Georgia Archives has a median lag of 107 years. Emory University's General Libraries: 137 years. The State Library of Pennsylvania, whose materials ended up in a Georgia aggregation for reasons worth investigating: 167 years. At the other end, Clark Atlanta University has a median lag of just 21 years, and the Museum of Contemporary Art of Georgia just 19.

These numbers don't mean some institutions were negligent and others were not. Digitization requires funding, staff, equipment, and rights clearances. An institution holding 19th-century manuscript collections faces different constraints than one documenting contemporary art. But the pattern of which institutions sit in which quadrant of this chart is not random.

04Who waited longest

Lag by institution type

Breaking the lag distribution down by institution type makes a quieter pattern visible. HBCU and Black cultural institutions tend to cluster in the middle range — not the extremes. But they are also working with older material than their median lag might suggest.

The institutions with the longest lags are concentrated in a specific category: rare book libraries and theological seminaries holding antebellum manuscript collections. This is a resource and mission story, not a negligence story. But it raises a question that the data can't answer: whose 19th-century materials got prioritized for early digitization funding, and whose waited?

06How this was made
Methodology & process notes ▶

The data source

All data comes from the DPLA bulk export for the Digital Library of Georgia hub, downloaded from the public S3 bucket s3://dpla-provider-export/2026/03/georgia/ using the AWS CLI. The export contains 1,149,945 records as gzipped JSONL files (~186MB compressed). No API key is required for bulk download access.

Processing

Records were processed locally with Python, reading each gzipped part file line by line. Key fields extracted:

  • sourceResource.date[0].begin — creation year (truncated to 4 digits)
  • sourceResource.type — format classification
  • sourceResource.subject[].name — subject headings
  • dataProvider.name — contributing institution
  • ingestDate — DPLA ingest date (used to compute digitization lag)
  • object — thumbnail URL for image display

Records with absent or unparseable dates were excluded. Items with computed lags over 200 years were excluded as likely metadata errors. A 3-year or 5-year rolling average was applied to annual counts to smooth noise from uneven cataloguing practices.

Role of Claude (Anthropic)

Claude assisted throughout this project in several ways — and ran into walls in several others, which turned out to be instructive.

What worked well: Claude helped draft and iterate the Python extraction scripts, suggested which fields to pull and how to handle edge cases in date parsing, proposed the lag analysis as a more interesting angle than simple subject frequency trends, helped design the visual encoding (scatter vs. beeswarm, what to put on each axis), wrote the D3.js visualization code, and helped compose the framing text for each section.

Where it hit walls: The original goal was to query the DPLA API live from the browser. This failed repeatedly due to CORS restrictions — the DPLA API does not return Access-Control-Allow-Origin headers, so browsers block direct client-side calls regardless of origin. A second attempt to route queries through the Anthropic API from the browser also failed for the same reason (the Anthropic API similarly blocks direct browser calls, as it is designed for server-side use). The bulk download was the right solution all along, and the CORS failures ultimately produced a richer dataset.

What this workflow is and isn't: Claude cannot run code, access the internet, or query APIs directly. It acts as a collaborator on code, structure, and analysis — the data processing, the curl of the S3 bucket, and the verification of results all happened on a local machine. The interpretive claims in this report emerged from a collaborative process — the author shaped the questions and Claude proposed framings, which the author accepted, modified, or rejected. Neither attribution feels complete.

Visualization

All charts are built with D3.js v7 and rendered as inline SVG in a standalone HTML file. No external data files are required — all processed data is embedded directly in the HTML. The report is designed to be shared as a single file.

Reproducibility

The DPLA bulk export is updated monthly. Running the extraction scripts against a later export will produce slightly different numbers as collections grow. The S3 path structure follows the pattern s3://dpla-provider-export/YYYY/MM/georgia/.jsonl/.
